Void Connect работает локально через mDNS, но может быть расширен до глобальной сети — соединяя пользователей из разных локальных сетей через интернет.

## Проблема: NAT

Домашний роутер имеет один внешний IP, за которым прячутся все устройства. Снаружи напрямую достучаться до конкретного устройства нельзя.

Торрент решил эту проблему — Void Connect использует те же подходы.

## Режимы работы

### Локальный (базовый)

Только mDNS, только LAN. Всё как описано в [[Обнаружение узлов]]. Ничего дополнительного не требуется.

### Расширенный (глобальный)

Те же узлы, но доступны из интернета. Данные по-прежнему живут на устройствах пользователей — децентрализованность не страдает.

## Выбранный подход: захардкоженные bootstrap-узлы

Именно так работают Bitcoin, Ethereum и большинство P2P сетей.

**Принцип**: сервер нужен только для первого знакомства узлов. После того как узлы нашли друг друга — сервер из цепочки полностью выпадает. Данные идут напрямую между узлами.

В код вшивается список публичных узлов сети:

```toml
# config.toml
[network]
bootstrap_nodes = [
  "node1.void-connect.example:7700",
  "node2.void-connect.example:7700",
]
```

Если один узел упал — клиент пробует следующий. Чем больше пользователей запустят публичный режим — тем надёжнее сеть.

### Публичный режим узла

Любой пользователь с белым IP или открытым портом может стать bootstrap-узлом:

```bash
void-connect --public
```

Такой узел добавляется в список известных точек входа и помогает другим найти сеть. Это и есть настоящая децентрализация — сеть принадлежит всем участникам.

## NAT Traversal: как узлы соединяются напрямую

После нахождения друг друга через bootstrap-узел, два клиента за NAT устанавливают прямое соединение.

Полный конвейер пробивания NAT в Void Connect состоит из четырёх шагов:

### 1. UPnP / NAT-PMP — открыть порт на роутере

Программа просит роутер автоматически пробросить рабочие порты (чанки/чат/DM/
сайты). Работает на большинстве домашних роутеров. **Это делает каждый узел** (не
только публичный) — best-effort, тихо игнорируя неудачу, если UPnP-шлюза нет.

### 2. STUN — узнать свой внешний адрес

Узел спрашивает у публичных STUN-серверов (Google, Cloudflare и др.) свой
внешний IP. Поднимать свой STUN не нужно. Используется публичными узлами для
рекламы достижимого извне адреса.

### 3. Публикация внешнего адреса через bootstrap (ключевой шаг)

Узел рекламирует свой **LAN**-адрес, но из другой сети он недостижим. Поэтому
bootstrap-узел при регистрации **подменяет приватный адрес узла на его внешний**
(источник TCP-соединения, как его видит bootstrap) и раздаёт уже его остальным.

В итоге другие узлы получают **внешний** `ip:port` и подключаются **напрямую** —
если этот порт открыт (шаг 1: UPnP / проброс / full-cone NAT). Раньше всем
раздавался LAN-адрес, и прямое соединение между сетями было невозможно — спасал
только relay. Теперь relay нужен реже.

> Bootstrap также делает **обратную пробу**: пытается сам подключиться к
> base-порту узла. Результат показывается в интерфейсе («порты открыты/закрыты»),
> чтобы пользователь понимал, доступен ли он напрямую.

### 4. Relay (fallback) — гарантированный путь

Если прямое соединение всё же не вышло (симметричный NAT без UPnP) — трафик идёт
через relay-сервер bootstrap-узла как ретранслятор. Это крайний случай; данные
всё равно зашифрованы E2E. Relay — **гарантия связности**: даже когда пробивание
не удалось, сообщения доходят.

> Полноценное «одновременное открытие» (simultaneous open) для симметричного NAT
> без UPnP — возможное развитие; сейчас этот случай покрывает relay.

## Что это не ломает

- **Децентрализованность**: данные живут на узлах, не на сервере
- **Неубиваемость**: bootstrap-узлов много, любой пользователь может добавить свой
- **Локальный режим**: LAN по-прежнему работает полностью без интернета
- **Безопасность**: см. [[Безопасность сети]] — шифрование не зависит от режима
