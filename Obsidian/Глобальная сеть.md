Void Connect работает локально через mDNS, но может быть расширен до глобальной сети — соединяя пользователей из разных локальных сетей через интернет.

## Проблема: NAT

Домашний роутер имеет один внешний IP, за которым прячутся все устройства. Снаружи напрямую достучаться до конкретного устройства нельзя.

Торрент решил эту проблему — Void Connect использует те же подходы.

## Режимы работы

### Локальный (базовый)

Только mDNS, только LAN. Всё как описано в [[Обнаружение узлов]]. Ничего дополнительного не требуется.

### Расширенный (глобальный)

Те же узлы, но доступны из интернета. Данные по-прежнему живут на устройствах пользователей — децентрализованность не страдает.

## Выбранный подход: захардкоженные bootstrap-узлы

Именно так работают Bitcoin, Ethereum и большинство P2P сетей.

**Принцип**: сервер нужен только для первого знакомства узлов. После того как узлы нашли друг друга — сервер из цепочки полностью выпадает. Данные идут напрямую между узлами.

В код вшивается список публичных узлов сети:

```toml
# config.toml
[network]
bootstrap_nodes = [
  "node1.void-connect.example:7700",
  "node2.void-connect.example:7700",
]
```

Если один узел упал — клиент пробует следующий. Чем больше пользователей запустят публичный режим — тем надёжнее сеть.

### Публичный режим узла

Любой пользователь с белым IP или открытым портом может стать bootstrap-узлом:

```bash
void-connect --public
```

Такой узел добавляется в список известных точек входа и помогает другим найти сеть. Это и есть настоящая децентрализация — сеть принадлежит всем участникам.

## NAT Traversal: как узлы соединяются напрямую

После нахождения друг друга через bootstrap-узел, два клиента за NAT устанавливают прямое соединение.

### UPnP / NAT-PMP

Программа просит роутер автоматически открыть порт. Работает на большинстве домашних роутеров.

### NAT Hole Punching

Самый элегантный метод. Оба узла одновременно отправляют пакеты друг другу через STUN-сервер. Роутеры с обеих сторон считают это исходящим трафиком и пропускают. В итоге — прямое соединение без посредника.

Для STUN используются **публичные бесплатные серверы** (Google, Cloudflare, открытые проекты) — поднимать свой не нужно.

### Relay (fallback)

Если hole punching не сработал (симметричный NAT) — трафик идёт через bootstrap-узел как ретранслятор. Это крайний случай, данные всё равно зашифрованы E2E.

## Что это не ломает

- **Децентрализованность**: данные живут на узлах, не на сервере
- **Неубиваемость**: bootstrap-узлов много, любой пользователь может добавить свой
- **Локальный режим**: LAN по-прежнему работает полностью без интернета
- **Безопасность**: см. [[Безопасность сети]] — шифрование не зависит от режима
