Общее хранилище позволяет всем участникам сети обмениваться файлами по принципу BitTorrent — каждый скачавший файл помогает его раздавать.

## Как это устроено

- Распределённое файловое хранилище на основе peer-to-peer обмена
- Файл разбивается на **чанки** (по 256 КБ)
- Каждый чанк имеет **SHA-256 хэш** — это его уникальный ID
- Узлы хранят у себя скачанные чанки и раздают их другим
- Индекс «кто какие чанки имеет» хранится в SQLite

## Процесс скачивания

1. Пользователь запрашивает файл
2. Сеть отвечает: «чанки X, Y, Z есть у узлов A, B, C»
3. Клиент качает чанки **параллельно с нескольких узлов**
4. После сборки проверяет целостность через SHA-256
5. Скачавший автоматически становится новым сидером (мульти-сидинг)

## Папки по пользователям (организация хранилища)

Чтобы один участник не мог «засорить» всё хранилище, а вредителя было видно
сразу, файлы в интерфейсе **сгруппированы по публикатору** — у каждого
пользователя своя «папка» с его именем.

- Каждая папка — это **исходный публикатор** файла (см. «Авторство» ниже).
- В заголовке папки видно **имя пользователя, число файлов и суммарный объём** —
  если кто-то залил гигабайты мусора, это сразу бросается в глаза.
- Своя папка («Вы») закреплена сверху и подсвечена.
- Папки сворачиваются/разворачиваются.

Такая организация даёт **подотчётность**: мусор и вредоносные файлы всегда
привязаны к конкретному пользователю, а не растворяются в общем списке.

### Анти-флуд (на будущее)

Визуальная группировка — первый рубеж. Поверх неё планируется **мягкая квота**:
ограничение суммарного объёма/числа файлов, которые узел принимает от одного
публикатора (манифесты сверх квоты от узла с низкой репутацией игнорируются).
Квота смягчается для узлов с высокой [[Система репутации|репутацией]].

## Авторство файла

У каждого файла навсегда сохраняется **исходный публикатор** — тот, кто первым
объявил файл в сети. Технически это поле `owner_key` (публичный ключ = ID
пользователя), которое записывается при первом появлении файла и больше не
перезаписывается (`INSERT OR IGNORE`), даже когда файл начинают раздавать другие
сидеры.

Зачем это нужно:

- **Поиск источника вредоносного контента.** Если кто-то опубликовал вирус —
  видно, в чьей папке он лежит и кто его автор. На такого можно подать жалобу.
- **Доверие.** Файл от пользователя с высокой репутацией надёжнее, чем от
  свежесозданного аккаунта с нулевой.

> Список *сидеров* (кто сейчас раздаёт чанки) — отдельный и может расти; но
> «папка»/автор всегда указывает на **первоисточник**.

## Поиск

Над списком есть строка поиска. Фильтрация идёт **по имени файла и по имени
публикатора** одновременно — можно быстро найти и конкретный файл, и все файлы
конкретного пользователя. Папки без совпадений скрываются, папки с совпадениями
раскрываются автоматически.

## Что вы раздаёте

Важно чётко понимать, какие файлы вы отдаёте в сеть. Поэтому:

- Файлы, которые **вы раздаёте** (опубликованные вами **или** полностью
  скачанные — вы автоматически становитесь сидером), помечены отдельным
  значком отдачи.
- Внизу страницы — счётчик «Раздаёте: N файлов».
- Статус каждого файла явный: *Ожидание / Загрузка / Пауза / Готово / Раздача*.

Так вы всегда видите, что именно лежит на вашем диске и отдаётся другим.

## Жалобы на публикатора

Из меню файла можно **пожаловаться на его публикатора** с указанием причины:

- Вредоносный контент (вирус/троян/майнер) → `MaliciousContent`
- Мошенничество / ложное описание → `MaliciousContent`
- Неверные метаданные / битые данные → `BadChunks`
- Другое

Жалоба подписывается вашим ключом и отправляется в [[Система репутации]]; при
накоплении жалоб от разных узлов публикатор получает штраф репутации и
автоматически изолируется. Пожаловаться на собственный файл нельзя.

## Целостность и защита

- Каждый чанк верифицируется по **SHA-256** после получения; несовпадение —
  чанк отбрасывается и запрашивается у другого узла, а отдавший плохой чанк
  получает штраф репутации.
- Авторство + жалобы + репутация вместе дают механизм борьбы с вредоносными
  файлами без центрального модератора.
